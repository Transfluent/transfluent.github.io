Transfluent is the only translation provider..

..to scale from 1 word into millions of words, yet providing excellent quality and having ability to translate into numerous of languages with very little effort from the customer.

Key to fulfill this promise is our innovative use of technology in the translation process. Our API is in the core enabling all this. On this website you will find detailed documentation of our API and description how to the translation process works.

Supported languages

Check supported language pairs from our website or call our API's languages method to list languages names and language pairs. The API methods can be called using ready-made clients or command line tools, for example call using curl: curl https://transfluent.com/languages/ or you can even do it with your browser: HTTP GET https://transfluent.com/languages/. The output is always JSON. As you can see, it is easy to get started with the API.

User accounts

To really get going, you will need a Transfluent user account. Either create a new user account by yourself or ask your company's account owner to create one for you.

Development and testing

While developing and testing your integration, you probably do not want to spend your own money for testing. Just for that, we provide a Pseudo language translation. The language code for pseudo language is xx-xx and the language id is 500. Use English (en-gb) as source language and Pseudo language as target language and the order will be processed for free. Instead of actual translation, you will get your source text reversed (e.g. "Hello" -> "olleH"). Placing Pseudo language orders makes sense for testing the whole process from end to end and depending on your use case possibly to pinpoint any non-translated parts on your end.

Payments

The easiest way to pay is by credit card. You or your company's account owner can setup a credit card in My Account. It is recommended to add a backup card so there will not be any interruption in the service if the primary card fails. If you prefer invoicing, please contact sales.

Use cases

  • API v2/text*: Plain text and HTML content. Content is stored by key (and group). Our backend keeps track of versions, so unchanged texts will not be re-translated.
  • API v2/file*: Resource files. Usually integrated to the application build process. Our backend keeps track of file versions, so unchanged parts will not be re-translated.
  • Translation Widget: ad-hoc translation needs, delivery by email. Text or files. Embedded version available for intranet.


RESTfulness

The API is RESTful. It reads and writes JSON

To Test the API visit the URI at https://transfluent.com/v2/hello/World which should show the result like

{% highlight javascript %} {"status":"OK","response":"Hello World"} {% endhighlight %}

You can change the last word in the link (Hello in this case) to something else, still the response will be same. Lets say change the world to my Name i.e Rajib, https://transfluent.com/v2/hello/Rajib Which should show the result like

{% highlight javascript %} {"status":"OK","response":"Hello Rajib"} {% endhighlight %}
Alternatively you can grab sample PHP client from GitHub's project page.

Authentication

Some methods are public and available without authentication. Some methods are available on Transfluent.com user session, e.g. CustomerEmail which is designed to serve the backend functionality via javascript to the customers. For developers, the most interesting methods are those which are available via token authentication. You can retrieve a token via Authenticate method.

Security

Transfluent's Backend API supports HTTPS for all methods and requires it for most of them. It is strongly encouraged to use HTTPS always.

Languages

Transfluent's Backend API uses internal language ids. See Languages method to retrieve them.

Errors

First check HTTP response code, next check response content. The API always returns a valid JSON response. On error the status is "ERROR".

{% highlight javascript %} {"status":"ERROR","error":{"type":"EBackendSecurityViolation","message":"Method requires secure connection (HTTPS), your request was unsafe."},"response":"Method requires secure connection (HTTPS), your request was unsafe."} {% endhighlight %}

It is recommended to match the error type (if non-generic type is provided) rather than trying to match the error text which may change for any reason (fixing typos, providing varying information like ids etc.)
